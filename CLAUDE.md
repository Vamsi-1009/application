# CLAUDE.md

Notes for whoever (human or Claude) picks this repo up next.

## What this repo is

A multi-page HTML "field manual" teaching how to build an application end-to-end — frontend, backend, database (structure, scaling, and attacks), auth/OTP, secrets management, email delivery, file storage/CDN, VMs, load balancers/autoscaling, domains, CI/CD, monitoring, security, and compliance (including India-specific: DPDP Act 2023, CERT-In). It is a teaching document, not a runnable app.

Landing page: [`docs/how-to-build-an-application.html`](docs/how-to-build-an-application.html) (hero + "how to use this manual" + a full directory of every stage). Each stage is its own HTML file (`docs/01-plan-and-threat-model.html` through `docs/19-reliability-and-disaster-recovery.html` — see README.md for the full list), sharing one CSS/JS design system generated from a single source of truth (see "Multi-page structure" below). Companion reference: [`docs/security/master_cybersecurity_checks_resources.html`](docs/security/master_cybersecurity_checks_resources.html) (a local copy of a 103-row security standards/tooling catalog the user supplied; Stage 17 explains and links every row individually instead of just embedding this file, and the manual's inline checklist items cite rows from it by standard name, e.g. `OWASP API Security · BOLA`, `CWE-89`).

## Multi-page structure — how it's generated

The manual was originally one long single-page HTML file, then split into one file per stage at the user's request ("frontend on the next page, backend next page, other things to the next page"). The split was done with a Node script (not committed to the repo — it was a one-off scratch tool) that:
1. Parsed the single-page source for its shared `<style>` block, the hero markup, and each `<section class="stage" id="sN">...</section>` block.
2. Wrote one HTML file per section into `docs/`, each with the full shared CSS/JS inlined (progress bar, scroll-reveal, back-to-top, TOC filter — same behavior as the original, minus the in-page scroll-spy since each page is now one section), and a prev/next footer nav computed from stage order.
3. The landing page (`how-to-build-an-application.html`) additionally gets the hero and a generated "full contents" grid linking every stage with its number and one-line dek.

**The left sidebar/rail nav was later removed entirely** at the user's explicit request ("i dont want this sidebar and info remove that") — every page is now full-width and centered (`.shell{ display:block; }`, `main{ margin:0 auto; }`). Cross-page navigation is only via the prev/next footer links and the landing page's contents grid. If re-adding any kind of persistent nav, confirm with the user first — it was deliberately taken out, not an oversight.

If asked to regenerate all pages from a single edited source, or to add a new stage: either hand-edit each affected file directly (they're independent, self-contained HTML files — no build step, no shared included files), or rewrite a similar one-off Node script if bulk-regenerating from a merged source is easier. There is no persistent build pipeline by design — these are meant to stay plain, dependency-free HTML files.

## History / how this ended up this way

The user's first message was ambiguous — it read as "build a full project (frontend, backend, deployment, VM, CDN, email/OTP, password wallet, load balancers, autoscaling, domains) that teaches the whole process." I asked clarifying questions, then built an actual working project ("PixelGram," an Instagram-style app) as three parallel agent builds: a Node.js/TypeScript/Express/Prisma/PostgreSQL backend, a React/TypeScript/Vite frontend, and Terraform + Nginx + systemd infra-as-code with a matching numbered tutorial doc set — all of it real, working code that type-checked and built cleanly.

The user then corrected: they wanted a **teaching page**, not a built project. I asked what to do with the generated code; the answer was **delete it** and produce **a single HTML page** instead. The `backend/`, `frontend/`, `infra/` directories and the app-specific tutorial docs (`docs/00-overview.md` through `docs/17-...md`, `docs/SECURITY.md`) were deleted. Only `docs/security/master_cybersecurity_checks_resources.html` (the user's own reference file, copied locally per their request) was kept, and the new standalone manual was written to replace everything else.

## Decisions baked into the manual's content

- Stack recommendations given in the manual (Node.js+TypeScript+Express or FastAPI for backend, React+TypeScript+Vite/Next.js for frontend, PostgreSQL, AWS as the cloud target) are **illustrative defaults for teaching purposes**, not a mandate — the doc says as much.
- No Docker/container content anywhere, per explicit user instruction (VMs are hardened and run the app via systemd directly).
- "Password wallet" is explicitly disambiguated in the manual (Stage 05) as **application secrets management** (AWS Secrets Manager / Vault for the app's own credentials), distinct from a per-user password-vault product feature, since the original phrase was ambiguous and most real apps mean the former.
- India-specific compliance content (DPDP Act 2023, CERT-In 6-hour incident reporting + log-retention directions, RBI rules for payments) was added in Stage 14 since the user asked for something usable "all over India."
- Design treatment (current): "living blueprint on carbon glass." Deep carbon ground (#060910), glassmorphic panels (backdrop-blur), aqua (#5eead4) primary + sky-blue (#4aa8ff) for data/gradients, amber (#ffb45e) retained for security boxes. Bold system-sans display headings (no webfont — keeps every file light and dodges the CSP font-CDN block), serif body, monospace labels/code. Signature motion: a live `<canvas id="bg-net">` network of drifting nodes + flowing data "packets" behind every page (the app assembling itself), a gradient count-up hero on the landing page, glass cards with pointer-driven 3D tilt, blur-in scroll reveals, animated accent edges on info boxes. All motion is gated by the base `prefers-reduced-motion` rule. Light + dark themes via CSS custom properties. (An earlier version used a flatter blueprint-navy palette with amber/teal — this replaced it on user request for a bolder UI.)
- The whole visual shell (CSS + JS + hero + canvas) is applied uniformly across all pages by a one-off "re-sheller" Node script (scratch tool, not committed): it extracts each page's `<title>`, `<nav class="rail">` (at the time it still existed), and `<main>` inner content, then re-wraps them in the shared new template — so restyling never touches the prose. The sidebar was removed in a later pass (see above). To restyle again, either re-run a similar re-sheller or hand-edit; there is no committed build step by design (plain dependency-free HTML).
- Eight pages (`01`, `02`, `03`, `04`, `05`, `07`, `09`, `19`) additionally have a small animated SVG "explainer figure" injected right under their intro paragraph (request flow, token refresh, middleware pipeline, cache-aside hit/miss, OTP lifecycle, CDN edge caching, load-balancing+autoscaling, database failover) — added via one-off scripts, not committed. All animation (the global pack + these figures + the background canvas) respects `prefers-reduced-motion`. The canvas additionally scales its node/packet count down on viewports under 600px for mobile performance.
- **Stage 19 (Reliability & resilience)** was added after the user pointed out specific missing topics: multiple servers, load balancers, backups, database replication, monitoring, auto-restart, redundancy, rate limiting, DDoS protection, disaster recovery. Several were already covered but scattered across other stages (replication in 04, auto-restart in 08, load balancing in 09, rate limiting in 03/06, monitoring in 12) — Stage 19 recaps those under a single "redundancy" framing and adds the two genuinely missing topics: DDoS protection (layered: CDN → managed DDoS service → WAF/rate limiting → autoscaling) and disaster recovery (RTO/RPO, multi-AZ vs multi-region, a tested DR runbook). It was appended at the end (not inserted mid-sequence) to avoid renumbering every other stage's filename/cross-references — same precedent as Stage 18.

## Site features (added after a "any suggestions?" pass)

- **SEO**: every page has a favicon (inline emoji SVG data URI — 🧭, no external asset), Open Graph + Twitter meta tags (title/description pulled from each page's own `<title>`/dek), and a `docs/sitemap.xml` + `docs/robots.txt`. Known limitation: those two files are served at `/application/sitemap.xml` and `/application/robots.txt`, not the domain root — this app doesn't own `webstocking.com`'s root (see Deployment below), so a crawler checking the conventional `/robots.txt` location won't find them. Acceptable trade-off given the constraint; flagged here so it isn't "fixed" by accident by touching the other app's territory.
- **Heading anchors**: every `<h3 class="reveal">` gets an auto-generated `id` (slugified from its text) and a hover-revealed `#` link, enabling deep-linking to any subsection.
- **Code copy buttons**: injected client-side via JS (`querySelectorAll('pre')`) rather than baked into each `<pre>`'s markup — simpler and can't drift out of sync with content edits. Captures `pre.innerText` *before* appending the button element, to avoid the button's own label text ending up in the copied string.
- **Site-wide search**: a floating button (+ Ctrl/Cmd-K) opens a modal listing all 20 pages, filtered live by title as you type. The page list (`SEARCH_INDEX`) is a static JSON array baked identically into every page at generation time — regenerate it (title + stage-num + filename per page) if a page's title changes or a new page is added.
- **Progress tracking**: each inner page has a "Mark this stage complete" toggle (`localStorage`, key `manual-progress`, keyed by filename). The landing page reads the same key on load to badge completed cards and show a completion count + progress bar. Purely client-side, no backend, resets if `localStorage` is cleared or a different browser/device is used — that's an accepted trade-off for staying a static, backend-free site.
- **Mini in-page TOCs**: pages `04`, `17`, `18`, `19` (the longest ones) get an "On this page" callout linking to that page's own heading anchors, inserted right after the stage's dek paragraph.

## Deployment

Live at **https://webstocking.com/application/** — an Ubuntu VM (user `vamsi`, non-default SSH port), Nginx, no containers, no build step (the `docs/` folder is served directly). Full setup, redeploy instructions, and the security hardening applied live in **[`docs/DEPLOY.md`](docs/DEPLOY.md)** — read that file before touching VM or deploy config, it's the source of truth, not this section.

Key facts worth knowing before assuming anything about the infra:
- **This VM hosts more than one project, developed in separate, uncoordinated sessions.** The domain `webstocking.com` already runs a separate, unrelated production app ("medical-scanner") at its root (proxying to `127.0.0.1:8000`) with its own Nginx server blocks and its own Let's Encrypt cert. This manual was added as a `location /application/` block *inside that existing Nginx vhost config* — it does not own the domain, does not have its own Nginx site file, and must never touch the other app's `/api/`, `/exposechain/`, or asset-proxy locations.
- **Auto-deploy**: `.github/workflows/deploy.yml` SSHes into the VM on every push to `main` and runs `git fetch && git reset --hard origin/main` inside the cloned repo at `/home/vamsi/Documents/application`. The SSH key GitHub Actions uses is a dedicated, **project-specifically-named** keypair (`~/.ssh/gha_deploy_application` on the VM, comment `github-actions-deploy-application`), restricted via a forced `command=` in `authorized_keys` so it can only ever run that exact deploy command — not get a shell.
- **A security pass was done on the live deployment** (headers, TLS, path traversal, `.git` exposure, reverse-tabnabbing links, third-party GitHub Action supply-chain pinning) — all fixed and verified; details and current state in `docs/DEPLOY.md`.
- **Incident, resolved: leaked private key.** While restricting the deploy key, an unrelated pre-existing private key (comment `webstocking-server`, origin unknown, predates this project) was found sitting in plaintext in `~/.ssh/authorized_keys` — a file that should only ever hold public keys. It was removed and the file rebuilt from scratch. That old key was also briefly pasted into a chat transcript during troubleshooting, so if it's ever traced to authorizing access somewhere else, it should be treated as compromised and rotated there too.
- **Incident, resolved: deploy key name collided with the "medical-scanner" project.** The other session on this VM generated its own deploy key using the same generic filename (`gha_deploy`) and `authorized_keys` comment (`github-actions-deploy`) we'd originally used, repeatedly overwriting our entry with its own (forced command `cd /home/vamsi/apps/Medical_scanner && bash deploy/deploy.sh`). Result: GitHub Actions reported green on every run while silently deploying the *other* project instead of this one — a green checkmark alone did not prove this repo's deploy actually ran. Fixed by renaming to a project-specific key (`gha_deploy_application` / `github-actions-deploy-application`), added as a new `authorized_keys` line without touching medical-scanner's, and updating the GitHub secret (had to **delete and re-add** the secret, not just "Update" — a bad paste is otherwise invisible). **Never reuse a generic deploy-key name or comment on this VM again** — always suffix with the project name. Full incident writeup in `docs/DEPLOY.md`.

## If asked to extend this

- Add new stages/sections directly in the relevant page file(s) in `docs/` — each is self-contained (inline CSS/JS, no build step, no external font/script CDNs). Adding a whole new stage means creating a new numbered file plus updating the landing page's contents grid and the neighboring pages' prev/next links.
- Any change to `docs/` on `main` deploys automatically (see Deployment above) — a bad push goes live within seconds, there's no staging environment or review gate beyond normal git/PR review if the user sets that up.
- If the user ever asks to actually *build* PixelGram (or any concrete app) again as real code, treat that as a new, separate request — confirm scope again before generating a full codebase, given the earlier scope mismatch in this project's history.
