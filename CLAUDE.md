# CLAUDE.md

Notes for whoever (human or Claude) picks this repo up next.

## What this repo is

A multi-page HTML "field manual" teaching how to build an application end-to-end — frontend, backend, database (structure, scaling, and attacks), auth/OTP, secrets management, email delivery, file storage/CDN, VMs, load balancers/autoscaling, domains, CI/CD, monitoring, security, and compliance (including India-specific: DPDP Act 2023, CERT-In). It is a teaching document, not a runnable app.

Landing page: [`docs/how-to-build-an-application.html`](docs/how-to-build-an-application.html) (hero + "how to use this manual" + a full directory of every stage). Each stage is its own HTML file (`docs/01-plan-and-threat-model.html` through `docs/18-your-first-build.html` — see README.md for the full list), sharing one CSS/JS design system generated from a single source of truth (see "Multi-page structure" below). Companion reference: [`docs/security/master_cybersecurity_checks_resources.html`](docs/security/master_cybersecurity_checks_resources.html) (a local copy of a 103-row security standards/tooling catalog the user supplied; Stage 21 explains and links every row individually instead of just embedding this file, and the manual's inline checklist items cite rows from it by standard name, e.g. `OWASP API Security · BOLA`, `CWE-89`).

## Multi-page structure — how it's generated

The manual was originally one long single-page HTML file, then split into one file per stage at the user's request ("frontend on the next page, backend next page, other things to the next page"). The split was done with a Node script (not committed to the repo — it was a one-off scratch tool) that:
1. Parsed the single-page source for its shared `<style>` block, the hero markup, and each `<section class="stage" id="sN">...</section>` block.
2. Wrote one HTML file per section into `docs/`, each with the full shared CSS/JS inlined (progress bar, scroll-reveal, back-to-top, TOC filter — same behavior as the original, minus the in-page scroll-spy since each page is now one section), and a prev/next footer nav computed from stage order.
3. The landing page (`how-to-build-an-application.html`) additionally gets the hero and a generated "full contents" grid linking every stage with its number and one-line dek.

**The left sidebar/rail nav was later removed entirely** at the user's explicit request ("i dont want this sidebar and info remove that") — every page is now full-width and centered (`.shell{ display:block; }`, `main{ margin:0 auto; }`). Cross-page navigation is only via the prev/next footer links and the landing page's contents grid. If re-adding any kind of persistent nav, confirm with the user first — it was deliberately taken out, not an oversight.

If asked to regenerate all pages from a single edited source, or to add a new stage: either hand-edit each affected file directly (they're independent, self-contained HTML files — no build step, no shared included files), or rewrite a similar one-off Node script if bulk-regenerating from a merged source is easier. There is no persistent build pipeline by design — these are meant to stay plain, dependency-free HTML files.

## History / how this ended up this way

The user's first message was ambiguous — it read as "build a full project (frontend, backend, deployment, VM, CDN, email/OTP, password wallet, load balancers, autoscaling, domains) that teaches the whole process." I asked clarifying questions, then built an actual working project ("PixelGram," an Instagram-style app) as three parallel agent builds: a Node.js/TypeScript/Express/Prisma/PostgreSQL backend, a React/TypeScript/Vite frontend, and Terraform + Nginx + systemd infra-as-code with a matching numbered tutorial doc set — all of it real, working code that type-checked and built cleanly.

The user then corrected: they wanted a **teaching page**, not a built project. I asked what to do with the generated code; the answer was **delete it** and produce **a single HTML page** instead. The `backend/`, `frontend/`, `infra/` directories and the app-specific tutorial docs (`docs/00-overview.md` through `docs/17-...md`, `docs/SECURITY.md`) were deleted. Only `docs/security/master_cybersecurity_checks_resources.html` (the user's own reference file, copied locally per their request) was kept, and the new standalone manual was written to replace everything else.

## Decisions baked into the manual's content

- Stack recommendations given in the manual (Node.js+TypeScript+Express or FastAPI for backend, React+TypeScript+Vite/Next.js for frontend, PostgreSQL, AWS as the cloud target) are **illustrative defaults for teaching purposes**, not a mandate — the doc says as much.
- No Docker/container content anywhere, per explicit user instruction (VMs are hardened and run the app via systemd directly).
- "Password wallet" is explicitly disambiguated in the manual (Stage 06) as **application secrets management** (AWS Secrets Manager / Vault for the app's own credentials), distinct from a per-user password-vault product feature, since the original phrase was ambiguous and most real apps mean the former.
- India-specific compliance content (DPDP Act 2023, CERT-In 6-hour incident reporting + log-retention directions, RBI rules for payments) was added in Stage 16 since the user asked for something usable "all over India."
- Design treatment (current): "living blueprint on carbon glass." Deep carbon ground (#060910), glassmorphic panels (backdrop-blur), aqua (#5eead4) primary + sky-blue (#4aa8ff) for data/gradients, amber (#ffb45e) retained for security boxes. Bold system-sans display headings (no webfont — keeps every file light and dodges the CSP font-CDN block), serif body, monospace labels/code. Signature motion: a live `<canvas id="bg-net">` network of drifting nodes + flowing data "packets" behind every page (the app assembling itself), a gradient count-up hero on the landing page, glass cards with pointer-driven 3D tilt, blur-in scroll reveals, animated accent edges on info boxes. All motion is gated by the base `prefers-reduced-motion` rule. Light + dark themes via CSS custom properties. (An earlier version used a flatter blueprint-navy palette with amber/teal — this replaced it on user request for a bolder UI.)
- The whole visual shell (CSS + JS + hero + canvas) is applied uniformly across all pages by a one-off "re-sheller" Node script (scratch tool, not committed): it extracts each page's `<title>`, `<nav class="rail">` (at the time it still existed), and `<main>` inner content, then re-wraps them in the shared new template — so restyling never touches the prose. The sidebar was removed in a later pass (see above). To restyle again, either re-run a similar re-sheller or hand-edit; there is no committed build step by design (plain dependency-free HTML).
- Nine pages (current numbers: `01`, `02`, `03`, `04`, `05`, `06`, `08`, `10`, `14`) additionally have a small animated SVG "explainer figure" injected right under their intro paragraph (request flow, pipes/redirection, token refresh, middleware pipeline, cache-aside hit/miss, OTP lifecycle, CDN edge caching, load-balancing+autoscaling, database failover) — added via one-off scripts, not committed. All animation (the global pack + these figures + the background canvas) respects `prefers-reduced-motion`. The canvas additionally scales its node/packet count down on viewports under 600px for mobile performance.
- **Reliability & resilience** was added (at the time, appended to the end as what was then "Stage 19") after the user pointed out specific missing topics: multiple servers, load balancers, backups, database replication, monitoring, auto-restart, redundancy, rate limiting, DDoS protection, disaster recovery. Several were already covered but scattered across other stages (replication in the database stage, auto-restart in the VM stage, load balancing in its own stage, rate limiting in the API/email stages, monitoring in its own stage) — this stage recaps those under a single "redundancy" framing and adds the two genuinely missing topics: DDoS protection (layered: CDN → managed DDoS service → WAF/rate limiting → autoscaling) and disaster recovery (RTO/RPO, multi-AZ vs multi-region, a tested DR runbook). It now lives at Stage 14, right before the security/compliance pre-launch pass — see the renumbering note below for why it moved from the end of the file list to there.
- **Your first build (the capstone stage, now Stage 18)** was added after a second "how do I make this the best way to actually learn to build and deploy" pass. Unlike the rest of the manual (reference material, read in any order), this stage is deliberately linear: a checkable build-order checklist (real testable outcomes, not "read stage X") and a capstone project spec ("TinyLink," an authenticated link shortener) designed so every one of its features maps back to a specific earlier stage. Checklist state persists via `localStorage` key `manual-checklist` (per-checkbox), separate from the per-page `manual-progress` key.
- **Linux fundamentals (now Stage 02)** was added after the user asked for terminal commands "from basic to the end" plus an explicit start/stop order across the whole manual. At the time it was filed last (append-only, to avoid renumbering) despite being conceptually a Stage-01.5 — see the renumbering note below for how it later actually moved to position 02. Content: the shell/filesystem tree, navigation, create/copy/move/delete (with an `rm -rf` `.mistake` box), permissions/ownership (its own `.security` checklist), pipes/redirection (with an animated SVG of `cat access.log | grep ERROR | sort | uniq -c`), text processing, processes/services, networking, package management, disk/archives, environment variables, cron, log reading, and a ~20-row cheat-sheet table. Small "New to the terminal commands here? See Stage 02" pointers live on the terminal-heavy stages (backend, database, VM, load balancing, CI/CD, monitoring, reliability), inserted as a muted line right after each stage's `.dek` paragraph.
- **The manual was later fully renumbered so file order equals reading order** — a reversal of the append-only-numbering policy described above (kept in this file as history, not as current policy). First pass: only a *display-only* layer was added (badges + recommended-order list showing the practice sequence while filenames stayed untouched, each badge carrying a small "(Stage NN)" sub-tag pointing back to the real file). The user then explicitly asked to "change the numbers of the boxes" and finally to physically rearrange the files themselves ("change the 21 to 01 or 02 or 03... place that at 3rd place") — at that point the tradeoff flipped: a teaching manual that's read start-to-finish benefits more from `01, 02, 03...` meaning exactly what it says than from stable-but-confusing legacy URLs, especially since this repo isn't yet linked from anywhere with real inbound traffic to protect. The actual renumbering (old→new): 01→01, 21→02, 02→03, 03→04, 04→05, 05→06, 06→07, 07→08, 08→09, 09→10, 10→11, 11→12, 12→13, 19→14, 13→15, 14→16, 18→17, 20→18, 15→19, 16→20, 17→21. Done with a one-off Python script (not committed, same "scratch tool" pattern as the original split/re-shell scripts): (1) two-phase file rename via temp names to dodge collisions in the permutation; (2) every `href`/`<loc>` filename reference rewritten across all 21 stage pages, the landing page, `sitemap.xml`, `README.md`, and the three `starter/` READMEs; (3) every inline "Stage NN" text mention site-wide renumbered via a single-pass regex substitution (avoids the chained-substitution bug a permutation like this would otherwise cause); (4) the embedded `SEARCH_INDEX` JSON's `"n"` fields renumbered and the array re-sorted into 01→21 order; (5) the now-redundant "(Stage NN)" sub-tags from the display-only pass stripped back out, since the real file number and the displayed number are identical again. The landing page's "recommended order" section and dek text were also rewritten from "the read order deviates from file order, here's why" framing to plain "just go straight through, 01 to 21." **If asked to add a new stage or reorder again**: there is no shared template, so repeat this same script-based approach (rename + global filename substitution + regex-based "Stage NN" substitution + SEARCH_INDEX resort) rather than hand-editing 20+ files one at a time.
- **Redirect stubs for the old, now-renumbered filenames** were added as a follow-up, since the physical rename above breaks any bookmark or external link to the old URLs (everything except Stage 01). Each old filename (e.g. `13-security-checklist.html`) now holds a small standalone page — not the real stage content — with `<meta http-equiv="refresh" content="0; url=NEW.html">`, a `<link rel="canonical">` pointing at the new URL, `<meta name="robots" content="noindex">`, and a `location.replace(...)` JS fallback. This is a client-side stand-in for a real 301 — good enough since old links land on the right content, but not as strong for SEO as a server-side redirect would be. A true 301 would need an Nginx `rewrite`/`return` rule added on the VM (outside this repo, see Deployment below) mapping each old filename to its new one; ask the user before doing that, since it touches server config directly rather than just `docs/`.

## Site features (added after two "any suggestions?" passes)

- **SEO**: every page has a favicon (inline emoji SVG data URI — 🧭, no external asset), Open Graph + Twitter meta tags (title/description pulled from each page's own `<title>`/dek), and a `docs/sitemap.xml` + `docs/robots.txt`. Known limitation: those two files are served at `/application/sitemap.xml` and `/application/robots.txt`, not the domain root — this app doesn't own `webstocking.com`'s root (see Deployment below), so a crawler checking the conventional `/robots.txt` location won't find them. Acceptable trade-off given the constraint; flagged here so it isn't "fixed" by accident by touching the other app's territory.
- **Heading anchors**: every `<h3 class="reveal">` gets an auto-generated `id` (slugified from its text) and a hover-revealed `#` link, enabling deep-linking to any subsection.
- **Code copy buttons**: injected client-side via JS (`querySelectorAll('pre')`) rather than baked into each `<pre>`'s markup — simpler and can't drift out of sync with content edits. Captures `pre.innerText` *before* appending the button element, to avoid the button's own label text ending up in the copied string.
- **Site-wide search**: a floating button (+ Ctrl/Cmd-K) opens a modal listing all 21 pages, filtered live by title as you type. The page list (`SEARCH_INDEX`) is a static JSON array baked identically into every page at generation time — regenerate it (title + stage-num + filename per page) if a page's title changes or a new page is added.
- **Progress tracking**: each inner page has a "Mark this stage complete" toggle (`localStorage`, key `manual-progress`, keyed by filename). The landing page reads the same key on load to badge completed cards and show a completion count + progress bar. Purely client-side, no backend, resets if `localStorage` is cleared or a different browser/device is used — that's an accepted trade-off for staying a static, backend-free site.
- **Difficulty/time badges**: a small pill (e.g. "~1.5 hrs · Intermediate") next to every stage's title, hardcoded per page (not computed) — update by hand if a stage's scope changes materially.
- **Mini in-page TOCs**: pages `02`, `05`, `14`, `17`, `21` (current numbers — the longest ones) get an "On this page" callout linking to that page's own heading anchors, inserted right after the stage's dek paragraph.
- **"Real mistake" callouts**: a new `.mistake` box style (danger-red accent, same visual family as `.callout`/`.security`/`.india`) documents two genuine incidents hit while building and deploying this exact site — the Nginx `sites-enabled` backup-file crash (Stage 09) and the SSH deploy-key naming collision that made GitHub Actions silently deploy the wrong project while still showing green (Stage 12). Authentic, not hypothetical — the CSS/JS is baked into all pages for reuse, but only these two currently have content.
- **Cumulative architecture diagram**: on the landing page's Stage 00 section, one animated figure showing the system's components appearing in build order (Browser → Load Balancer → Backend → Database → Storage/CDN → Monitoring), each staying "lit" once activated rather than fading out — reinforces that this is one system built incrementally, distinct from the seven per-stage explainer figures which each illustrate a single isolated concept.

## The starter repo (`starter/`)

After the second "any suggestions?" pass, the user explicitly asked for a real runnable companion codebase (a reversal of the earlier "teaching page, not a built project" decision from this repo's history — confirmed deliberately this time via an explicit question, not assumed). Scoped down to exactly the Stage 18 capstone (TinyLink, a link shortener) rather than another Instagram-style clone, since it's small enough to actually finish and still touches auth/OTP, ownership checks, rate limiting, and the DB/frontend patterns from Stages 03-07 and 17.

**Lives at `starter/`, sibling to `docs/`, not inside it** — the VM's Nginx config only serves `docs/`, so nothing here can accidentally go live by being in the wrong folder.

- `starter/backend/` — Node.js + TypeScript + Express + Prisma + PostgreSQL. `npm install` (0 vulnerabilities after bumping `nodemailer` to `^9.0.3` and `vitest` to `^4.1.10` past flagged CVEs from what were initially more "obvious" version picks), `npm run typecheck`, `npm run build`, and `npm test` all verified passing before commit — a real bug was caught and fixed in this pass (`jwt.SignOptions["expiresIn"]` type mismatch with the zod-validated env string) and a real test-setup gap (`vitest.config.ts` needed dummy env vars so `config/secrets.ts`'s intentional fail-fast validation doesn't block test runs).
- `starter/frontend/` — React + TypeScript + Vite. Same install/typecheck/build verification. Implements the exact in-memory-token + refresh-on-401 pattern Stage 03 teaches, not a simplified version of it.
- If asked to extend the starter (e.g. the per-link preview image mentioned in the Stage 18 capstone brief), treat it like any other real codebase change: run `npm run typecheck`/`build`/`test` before considering it done, the same rigor already applied here.
- **A full security review of `starter/` was done** (two independent audits, backend and frontend) after the manual renumbering work above. Overall verdict: solid as a teaching example — Prisma parameterized queries throughout (no raw SQL), the delete-link ownership/IDOR check is correct, OTP is `crypto.randomInt` + Argon2id-hashed with attempt limits, JWTs are `HS256`-pinned with rotating/hashed refresh tokens, CORS is an explicit allow-list, and every route uses `.strict()` Zod schemas. Two real gaps were found and fixed:
  - `links.schemas.ts` — `targetUrl` only checked `z.string().url()`, so the redirect endpoint (`GET /:code` → `res.redirect(302, targetUrl)`) accepted `javascript:`/`file:` schemes and loopback/private/cloud-metadata IPs (`127.0.0.1`, `169.254.169.254`, etc.) — an open-redirect/SSRF-adjacent hole. Added `isAllowedTargetUrl()` (scheme allow-list + private/loopback IP denylist, no new dependency) with a dedicated test file (`tests/links.schemas.test.ts`).
  - `errorHandler.ts` — the prod/dev branch failed **open toward leaking**: raw error details were redacted only when `NODE_ENV === "production"` exactly, so an unset or misconfigured `NODE_ENV` in a real deploy would leak stack traces. Flipped to redact-by-default, only showing raw errors when `NODE_ENV === "development"` explicitly.
  - The frontend's `shortUrl` (rendered as an `<a href>` on the dashboard) is **not** attacker-controlled — it's always `${PUBLIC_BASE_URL}/${server-generated code}`, never the raw `targetUrl` — so no separate frontend fix was needed there; `targetUrl` itself is only ever rendered as plain text, not as an `href`.
  - Minor, not fixed (logged here in case a future pass wants them): dev-mode OTP-to-console fallback (`lib/mailer.ts`) doesn't hard-fail if `NODE_ENV=production` and SMTP is unconfigured — it silently degrades instead; short-code brute-force is rate-limited per-IP only, not globally, so a distributed attempt could still enumerate codes over time (impractical given the 33^7 keyspace, but not impossible).

## Deployment

Live at **https://webstocking.com/application/** — an Ubuntu VM (user `vamsi`, non-default SSH port), Nginx, no containers, no build step (the `docs/` folder is served directly). Full setup, redeploy instructions, and the security hardening applied live in **[`docs/DEPLOY.md`](docs/DEPLOY.md)** — read that file before touching VM or deploy config, it's the source of truth, not this section.

Key facts worth knowing before assuming anything about the infra:
- **This VM hosts more than one project, developed in separate, uncoordinated sessions.** The domain `webstocking.com` already runs a separate, unrelated production app ("medical-scanner") at its root (proxying to `127.0.0.1:8000`) with its own Nginx server blocks and its own Let's Encrypt cert. This manual was added as a `location /application/` block *inside that existing Nginx vhost config* — it does not own the domain, does not have its own Nginx site file, and must never touch the other app's `/api/`, `/exposechain/`, or asset-proxy locations.
- **Auto-deploy**: `.github/workflows/deploy.yml` SSHes into the VM on every push to `main` and runs `git fetch && git reset --hard origin/main` inside the cloned repo at `/home/vamsi/Documents/application`. The SSH key GitHub Actions uses is a dedicated, **project-specifically-named** keypair (`~/.ssh/gha_deploy_application` on the VM, comment `github-actions-deploy-application`), restricted via a forced `command=` in `authorized_keys` so it can only ever run that exact deploy command — not get a shell.
- **A security pass was done on the live deployment** (headers, TLS, path traversal, `.git` exposure, reverse-tabnabbing links, third-party GitHub Action supply-chain pinning) — all fixed and verified; details and current state in `docs/DEPLOY.md`.
- **Incident, resolved: leaked private key.** While restricting the deploy key, an unrelated pre-existing private key (comment `webstocking-server`, origin unknown, predates this project) was found sitting in plaintext in `~/.ssh/authorized_keys` — a file that should only ever hold public keys. It was removed and the file rebuilt from scratch. That old key was also briefly pasted into a chat transcript during troubleshooting, so if it's ever traced to authorizing access somewhere else, it should be treated as compromised and rotated there too.
- **Incident, resolved: deploy key name collided with the "medical-scanner" project.** The other session on this VM generated its own deploy key using the same generic filename (`gha_deploy`) and `authorized_keys` comment (`github-actions-deploy`) we'd originally used, repeatedly overwriting our entry with its own (forced command `cd /home/vamsi/apps/Medical_scanner && bash deploy/deploy.sh`). Result: GitHub Actions reported green on every run while silently deploying the *other* project instead of this one — a green checkmark alone did not prove this repo's deploy actually ran. Fixed by renaming to a project-specific key (`gha_deploy_application` / `github-actions-deploy-application`), added as a new `authorized_keys` line without touching medical-scanner's, and updating the GitHub secret (had to **delete and re-add** the secret, not just "Update" — a bad paste is otherwise invisible). **Never reuse a generic deploy-key name or comment on this VM again** — always suffix with the project name. Full incident writeup in `docs/DEPLOY.md`.

## If asked to extend this

- Add new stages/sections directly in the relevant page file(s) in `docs/` — each is self-contained (inline CSS/JS, no build step, no external font/script CDNs). Adding a whole new stage means creating a new numbered file plus updating the landing page's contents grid and the neighboring pages' prev/next links.
- Any change to `docs/` on `main` deploys automatically (see Deployment above) — a bad push goes live within seconds, there's no staging environment or review gate beyond normal git/PR review if the user sets that up.
- If the user ever asks to actually *build* PixelGram (or any concrete app) again as real code, treat that as a new, separate request — confirm scope again before generating a full codebase, given the earlier scope mismatch in this project's history.
