A backend, file by file — then an AI security audit

Everything above has been explained concept-first. This stage does it code-first: one concrete backend file tree, what each file actually does and why it earns its place, then the exact process for having an AI (or a human) audit the finished code against the catalog above.

Who wrote this example, and why that's disclosed

The file tree and snippets below are illustrative teaching code, drafted with an AI pair-programmer (in this case, Claude) rather than hand-typed line by line. That's disclosed here on purpose: whether code comes from a human, an AI assistant, or both, the rule doesn't change — every line that reaches production must be read, understood, and reviewed by someone accountable for it. AI-assisted code is not exempt from review; if anything it needs the same scrutiny you'd give a contractor's first pull request.

The file tree

A realistic shape for the Node.js + TypeScript + Express + PostgreSQL/Prisma backend described in Stages 03–07, applied to a photo-sharing app like the one in this manual's examples:

backend/
  src/
    app.ts                  # builds the Express app: middleware chain + routes
    server.ts               # boots the HTTP server; loads/validates secrets first
    config/
      secrets.ts            # reads & validates env vars, fails fast if any are missing
      logger.ts             # structured logging, redacts secrets/tokens automatically
    middleware/
      auth.ts               # verifies the JWT, attaches req.user
      validate.ts           # runs a zod schema against body/params/query
      rateLimit.ts           # throttles auth/OTP endpoints
      errorHandler.ts        # catches errors, hides internals in production
    routes/
      auth.routes.ts         # /api/auth/*  — signup, login, OTP, refresh
      posts.routes.ts        # /api/posts/* — create, delete, like, comment
      feed.routes.ts         # /api/feed     — paginated timeline
    modules/
      auth/                  # controller + service: the actual signup/login logic
      posts/                 # controller + service: the actual post logic
    lib/
      prisma.ts              # the one shared database client
      jwt.ts                 # sign/verify tokens, explicit algorithm allow-list
      password.ts            # Argon2id hash/verify wrapper
      mailer.ts              # sends OTP/reset emails via SMTP
      s3.ts                  # issues presigned upload URLs
  prisma/
    schema.prisma            # the database schema, version-controlled
  .env.example                # every required secret NAME, no real values
  package.json                # pinned dependency versions

What each file does, and the benefit of splitting it out this way

FileWhat it doesWhy it's worth the separate file
app.tsCreates the Express app and wires every piece of middleware and every route into one place, in a fixed order.One file is the single source of truth for "what runs on every request." A security review only has to read this one file to know the whole request pipeline.
server.tsImports app.ts, loads and validates secrets, then starts listening on a port.Separated from app.ts so the app itself can be imported into tests without actually opening a network port.
config/secrets.tsParses process.env through a strict schema; throws immediately if anything required is missing or malformed.Turns "the app is silently misconfigured in production" into "the app refuses to start" — a much cheaper failure to catch.
config/logger.tsA shared logger that automatically strips Authorization headers, passwords, and OTP codes before writing a log line.Without a shared logger, it's easy for one route to accidentally console.log(req.body) and leak a password into a log file.
middleware/auth.tsReads the JWT from the Authorization header, verifies its signature and algorithm, and attaches the decoded user to req.user.Every protected route reuses this instead of re-implementing token verification — one bug fixed here fixes it everywhere.
middleware/validate.tsTakes a zod schema and rejects the request with 400 if the body/params/query don't match it exactly.Centralizes input validation so a route handler can trust its input has the right shape before it touches the database.
middleware/rateLimit.tsCounts requests per account/IP and returns 429 past a threshold, tuned per endpoint (tighter on OTP than on the feed).One tunable place to slow down brute-force and abuse instead of scattering counters through business logic.
middleware/errorHandler.tsThe last middleware in the chain — catches anything thrown upstream and returns a generic message in production, full detail only in server-side logs.Guarantees no route can accidentally leak a stack trace, SQL error, or file path to a client, even one nobody remembered to wrap in a try/catch.
routes/*.tsMaps a URL + HTTP verb to a handler, and lists which middleware guard that specific route (e.g. requireAuth, then validate(schema), then the handler).Reading one route file tells you exactly what protects that endpoint — nothing hidden in the handler itself.
modules/*/service.tsThe actual business logic: "create this post," "follow this user" — including the ownership/authorization check (is this my post to delete?).Keeps the authorization decision physically next to the operation it guards, instead of trusting a route comment to remind someone.
lib/jwt.tsOne function to sign a token, one to verify it, with the signing algorithm hardcoded to a single allowed value.Stops a developer three months from now from accidentally accepting a token signed with a weaker or attacker-chosen algorithm.
lib/password.tsWraps Argon2id so the rest of the codebase never touches a hashing function directly.If the hashing algorithm ever needs to change, it changes in one file, not in every place a password is checked.
lib/s3.tsIssues a short-lived, scoped presigned upload URL for a given user's own object-key prefix.Guarantees the API server itself never receives or stores raw image bytes — removes a whole class of upload-based attacks from its surface.
prisma/schema.prismaDeclares every table, column, index, and uniqueness constraint as version-controlled text.Every schema change is a reviewable diff and a migration file, not an unrecorded click in a database GUI.
.env.exampleLists every secret name the app needs, with placeholder (fake) values.A new developer knows exactly what to configure without ever being handed a real credential over Slack or email.

CORS, exactly where it lives and what it actually decides

// inside app.ts — this runs before any route sees the request
app.use(cors({
  origin(origin, callback) {
    // "origin" is the website that sent this request (from the browser's
    // Origin header). Undefined means it's not a browser request at all
    // (curl, a mobile app, server-to-server) — allow those through.
    if (!origin || ALLOWED_ORIGINS.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error("Origin not allowed by CORS policy"));
    }
  },
  credentials: true, // allows the browser to send the httpOnly refresh cookie
}));

CORS is a browser-enforced rule about which websites are allowed to read the response of a cross-origin request. Get it wrong in either direction and something breaks: set origin: "*" with credentials: true and browsers will refuse it outright (and if it did work, it would let any website on the internet make authenticated requests using a logged-in user's cookie); lock it down too tightly and your own frontend can't call your own API. The fix is always the same: an explicit allow-list of exactly the origins that should be able to call you — your production frontend domain, your local dev server, nothing else.

Middleware order is itself a security decision

// app.ts, in this exact order — the order is not arbitrary:
app.use(helmet());               // 1. security headers on every response, first
app.use(cors(corsOptions));      // 2. reject disallowed origins before anything else runs
app.use(express.json({ limit: "100kb" })); // 3. parse body, but cap its size
app.use(cookieParser());         // 4. read the refresh-token cookie
app.use(requestLogger);          // 5. log the request (after body parsing, so size is known)

app.use("/api/auth", authRoutes);   // 6. routes — each internally adds requireAuth/
app.use("/api/posts", postsRoutes); //    validate/rateLimit where that route needs it
app.use("/api/feed", feedRoutes);

app.use(notFoundHandler);        // 7. anything unmatched becomes a clean 404
app.use(errorHandler);           // 8. MUST be last — Express only treats a 4-arg
                                  //    function as an error handler, and only errors
                                  //    thrown by earlier middleware reach it

If errorHandler moved earlier, thrown errors from routes registered after it would never be caught. If cors moved after the routes, a disallowed origin's request would already have executed your business logic before being rejected. The order in the file is the security model — which is exactly why Stage 03's authorization check inside a route handler is a second, independent layer, not a replacement for getting this order right.

Routes: guards stacked in front of the handler

// routes/posts.routes.ts
router.delete(
  "/:id",
  requireAuth,                 // 1. must be logged in at all
  validate(idParamSchema, "params"), // 2. :id must actually look like a valid id
  asyncHandler(deletePostHandler)   // 3. only now does business logic run,
);                                   //    which itself checks post.authorId === req.user.id

Read a route top to bottom as a checklist of what must be true before the handler runs. Delete any one guard and ask what breaks: no requireAuth — anyone, logged in or not, can call this; no validate — a malformed :id reaches the database layer instead of failing cleanly; and even with both, the handler itself still has to check ownership, because "logged in" and "owns this specific post" are two different questions (this is the BOLA/IDOR check from Stage 03).

.env — where the app's own passwords actually live

# .env.example — committed to git, values are placeholders only
DATABASE_URL=postgres://user:CHANGE_ME@localhost:5432/pixelgram
JWT_SIGNING_KEY=replace_with_a_random_32+_character_value
SMTP_HOST=email-smtp.ap-south-1.amazonaws.com
SMTP_USER=replace_with_ses_smtp_username
SMTP_PASSWORD=replace_with_ses_smtp_password
AWS_ACCESS_KEY_ID=replace_me
AWS_SECRET_ACCESS_KEY=replace_me
S3_BUCKET=pixelgram-media

# .env — NEVER committed (this exact filename is in .gitignore), holds
# the REAL values above, used only for local development.
Why this file gets called a "password wallet"
  • .env.example is safe to commit — it documents which secrets exist without exposing any of them CWE-798
  • The real .env lives only on a developer's machine, listed in .gitignore, never pushed CWE-798
  • In production, none of this comes from a file at all — config/secrets.ts reads environment variables that the platform injects at deploy time from a real secrets manager (Stage 05/13: AWS Secrets Manager, Vault) CSA Cloud Controls Matrix
  • If a real .env is ever committed by accident, treat every value in it as compromised — rotate all of them, don't just delete the file from a later commit (git history still has it) NIST SSDF

The code is "done." Now audit it.

Passing your own manual testing is not the same as being secure — it only proves the features work when used as intended. The next step is deliberately trying to break your own rules, checking the finished code against named standards instead of vibes. Here's what that looks like against the file tree above, tied directly to entries from Stage 17's catalog:

CheckWhere to lookStandard
Does every route that reads/writes a specific resource verify the caller owns it, not just that they're logged in?modules/*/service.ts — every function taking an idOWASP API Security · BOLA
Is any SQL built by string concatenation instead of going through Prisma's parameterized queries?grep the whole src/ tree for raw $queryRaw usageCWE-89
Is there a hardcoded credential, API key, or secret anywhere in source instead of read from config/secrets.ts?grep for suspicious literals: key=, password=, AWS key patternsCWE-798
Does the error handler ever return a stack trace, SQL error text, or internal file path to the client?middleware/errorHandler.tsOWASP ASVS
Are auth, OTP-send, OTP-verify, and password-reset all rate-limited, per account and per IP?routes/auth.routes.ts, middleware/rateLimit.tsCWE-307
Does the JWT verification step pin an explicit algorithm, rejecting "none" or an attacker-chosen alg?lib/jwt.tsOWASP · Broken Authentication
Is CORS an explicit origin allow-list rather than * or a reflected Origin header?app.tsCWE-942
Do any dependencies in package.json have a known high-severity advisory?npm audit, or paste package.json into OSV.devOSV / GitHub Advisory Database

Turning that into a prompt you actually hand an AI

This is the literal mechanism the question is asking about: you give an AI coding assistant (Claude, or any other) the standards to check against and the codebase to check them in, and ask it to report concrete findings instead of general advice. A prompt that actually works, adapted from the table above:

Perform a security review of the backend in src/. For each finding,
report the exact file and line, the vulnerable pattern, which standard
it violates, and the minimal fix — no generic advice, no rewording the
standard back at me.

Check specifically for:
1. Missing ownership/authorization checks on any route or service
   function that reads or mutates a specific resource by id
   (OWASP API Security — Broken Object Level Authorization).
2. Any raw/string-built SQL instead of parameterized queries (CWE-89).
3. Hardcoded secrets, API keys, or credentials in source instead of
   read from the environment (CWE-798).
4. Error responses that leak stack traces, SQL text, or file paths in
   production (OWASP ASVS).
5. Auth, OTP, and password-reset endpoints missing rate limiting (CWE-307).
6. JWT verification that doesn't pin an explicit signing algorithm.
7. A CORS policy that isn't an explicit origin allow-list.

Reference material for the above:
- OWASP API Security Top 10 — https://owasp.org/www-project-api-security/
- MITRE CWE — https://cwe.mitre.org/
- OWASP ASVS — https://owasp.org/

List every finding, ranked most severe first. If a check passes, say so
explicitly rather than omitting it — I need to know what was verified,
not just what failed.

Two things make this work better than "review my code for security issues": naming the exact standards (so the AI checks against a real bar, not a vague sense of "best practice"), and demanding file:line findings with a stated pass/fail per check (so silence doesn't get mistaken for "nothing was found"). This is precisely what a /security-review-style workflow automates — feed it your diff or codebase, point it at the catalog from Stage 17, and read the findings before you ship.

What an AI review is, and isn't

An AI pass like this is fast, cheap, and good at catching the exact patterns above — because they're well-documented, named vulnerability classes it has seen thousands of times. It's not a substitute for npm audit/OSV (which know about specific CVEs in your exact dependency versions), and it's not a substitute for a real penetration test before handling sensitive data at scale (Stage 14's PTES/OSSTMM/CREST entries). Use it as a fast first pass on every change, not the only pass before launch.

Two findings this exact prompt would catch

Before — vulnerable

async function deletePost(req, res) {
  await db.post.delete({ where: { id: req.params.id } });
  res.status(204).end();
}

Any logged-in user can delete any post by guessing its id — no ownership check. OWASP API Security · BOLA

After — fixed

const post = await db.post.findUnique({ where: { id } });
if (post.authorId !== req.user.id) return res.status(403).end();
await db.post.delete({ where: { id } });

One extra check, one extra line — the entire class of bug, closed.
