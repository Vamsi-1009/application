Full resource catalog, explained

All 103 entries from the source catalog, grouped by what they're actually for, each with a plain-language explanation and a direct link — not just a reprint of the raw table. This is the reference every STD tag earlier in this manual points back to.

Read this as a map, not a to-do list: most single apps only need a handful of these seriously (Stage 13's consolidated checklist already pulled those out). The rest exist so that when you eventually touch mobile, AI features, IoT, or a regulated industry, you know exactly where to look instead of starting from zero.

1 · Application, API & mobile security testing

Where you go to learn how to find and fix bugs in the app you're building right now — the most directly useful group for Stages 02–07 above.

#ResourceWhat it's for
1OWASPThe umbrella project behind the Top 10, ASVS (a checklist of security requirements), WSTG (a testing guide), and dozens of cheat sheets. Start here for almost any web security question.
2PortSwigger Web Security AcademyFree, hands-on labs that let you actually exploit SQLi, XSS, CSRF, SSRF, JWT flaws, and access-control bugs in a safe sandbox — the fastest way to build real intuition.
3OWASP API Security Top 10The API-specific version of the Top 10 — BOLA/IDOR (Stage 03's authorization checks), broken auth, and unrestricted resource consumption all come from this list.
4OWASP MAS (Mobile)The mobile equivalent — relevant the moment PixelGram-style app grows a native iOS/Android client instead of just a web frontend.
5OWASP GenAI Security ProjectRisks specific to features built on LLMs — prompt injection, insecure output handling, excessive agency. Relevant only if you add an AI feature.
6OWASP AI ExchangeA broader knowledge base of AI threats and controls, one level up from the GenAI-specific project above.
7OWASP Firmware Security Testing MethodologyOnly relevant if your product includes an embedded device — extraction, binary analysis, and credential-hunting on firmware images.
8OWASP Software Component Verification StandardA checklist for vetting third-party/open-source packages before you depend on them — pairs with Stage 11's dependency scanning.
9OWASP SAMMA maturity model for your whole engineering org's security practice — useful once you're past a single app and building a security program.

2 · Vulnerability & threat intelligence

Where known bugs, exploited-in-the-wild flaws, and severity scores live — what you'd check before shipping a dependency, and what an incident-response process (Stage 12) draws on.

#ResourceWhat it's for
10MITRE CWEThe dictionary of weakness types (CWE-89 SQL injection, CWE-798 hardcoded credentials, etc.) — this is what every CWE-### tag in this manual refers to.
11CAPECCatalogs how a weakness gets exploited in practice — the attack-pattern layer above CWE's weakness layer.
12CVEThe canonical ID system for a specific disclosed vulnerability in a specific product — "CVE-2024-xxxxx" always refers to CVE.org's record.
13NIST NVDEnriches raw CVEs with severity scores and affected-version data — where you'd look up how bad a given CVE actually is.
14CISA KEVA shortlist of vulnerabilities known to be actively exploited — patch these before anything else on a CVE list.
15Google OSVVersion-aware vulnerability lookup for open-source packages — feed a package name + version, get back known issues. Good to wire into CI.
16GitHub Advisory DatabaseWhat powers npm audit and Dependabot alerts — advisories across npm, pip, Maven, Go, and more.
17CERT/CC Vulnerability NotesTechnical write-ups and coordinated-disclosure notes for individual vulnerabilities, with remediation guidance.
18VulDBA commercial vulnerability tracking and intelligence database, broader than the free government feeds.
19VulnCheckExploitation intelligence aimed at helping you prioritize which vulnerabilities to patch first based on real-world exploitation signals.
20Exploit DatabaseA public archive of proof-of-concept exploits — use defensively, to confirm whether a system you're authorized to test is actually exposed.
21FIRST CVSSThe standard formula behind a vulnerability's numeric severity score (0–10) that most vulnerability databases quote.
22FIRST EPSSEstimates the probability a given CVE will actually be exploited in the next 30 days — a sharper prioritization signal than severity alone.
23FIRSTThe umbrella organization coordinating CSIRTs/incident-response teams globally — relevant once you have a real incident-response process (Stage 12).

3 · Hardening baselines & governance frameworks

The "how do I configure this server/cloud account/org correctly" layer — directly behind Stage 08's VM hardening and Stage 01's threat-modeling.

#ResourceWhat it's for
24CIS BenchmarksStep-by-step hardening baselines for Linux, Windows, SSH, databases, Kubernetes, and cloud services — the source behind Stage 08's SSH/firewall checklist.
25CIS ControlsA prioritized, higher-level list of safeguards (asset inventory, access control, logging, recovery) — good for deciding what to do first with limited time.
26CIS Controls NavigatorA tool for mapping CIS Controls onto other frameworks you might already be tracking (ISO 27001, NIST CSF, etc.).
27DISA STIGsVery detailed, US-military-grade hardening requirements — more rigorous than most startups need, but a good reference for maximum-assurance configuration.
28OpenSCAPA tool that automatically checks a machine's configuration against a hardening baseline (like a CIS Benchmark) instead of doing it by hand.
29SCAPThe underlying machine-readable standard that tools like OpenSCAP implement.
30NIST Cybersecurity FrameworkA five-function structure (Govern, Identify, Protect, Detect, Respond, Recover) for organizing a security program end-to-end — referenced throughout Stage 12 and 14.
31NIST SSDFSecure software development practices — the source behind Stage 11's "scan dependencies, sign builds" CI/CD guidance.
32NIST SP 800-207 (Zero Trust)The formal definition of "zero trust architecture" — never implicitly trust a request just because it's inside your network perimeter.
33NIST Cryptographic StandardsWhich hashing/encryption algorithms are actually considered safe today — the basis for Stage 05's Argon2id recommendation.
34NIST AI RMFGovernance and risk-management guidance specifically for AI systems — relevant if you ship an AI-powered feature.
35MITRE ATT&CKA knowledge base of real attacker tactics and techniques (how they gain persistence, escalate privileges, steal credentials) — the basis for Stage 01's threat-modeling and Stage 12's "watch for credential stuffing" alert.
36MITRE ATLASATT&CK's counterpart specifically for attacks against AI/ML systems (model theft, data poisoning, prompt injection).
37MITRE D3FENDA companion knowledge base of defensive techniques, organized to map directly against ATT&CK's attacker techniques.
38Center for Threat-Informed DefenseResearch connecting real observed attacker behavior to concrete defensive mappings — a practical layer on top of ATT&CK.
39Lockheed Martin Cyber Kill ChainAn older but still-useful model of the stages of an intrusion (recon → exploit → command-and-control → exfiltrate) for designing layered defenses.

4 · Cloud, containers & software supply chain

Directly behind Stages 08–11 — VM/cloud configuration, and making sure the code and dependencies you deploy weren't tampered with on the way.

#ResourceWhat it's for
40Cloud Security AllianceCloud-specific guidance on IAM, data protection, and governance — the general umbrella for the cloud-focused entries below.
41CSA Cloud Controls MatrixA detailed control framework for cloud environments — the source behind Stage 04/09's "private subnets, security-group scoping" guidance.
42CSA STARA registry where cloud providers publish self-assessments and certifications of their security posture — useful for vendor due diligence.
43FINOS Common Cloud ControlsCloud control definitions aimed at regulated financial environments — relevant if PixelGram-style app ever adds payments/wallets.
44CNCFThe foundation behind Kubernetes and the broader cloud-native ecosystem — relevant if you later move from plain VMs to container orchestration.
45Kubernetes Security DocumentationRBAC, secrets, and workload-security guidance specifically for Kubernetes — not used in this manual since it deliberately skips containers, but the reference to have on hand if you adopt them later.
46OpenSSFAn industry-wide initiative on open-source security practices — dependency scanning, build security, and the source behind Stage 11's CI guidance.
47SLSAA framework for build provenance — proving that what's deployed is exactly what was reviewed, not a tampered artifact. Referenced in Stage 11.
48SigstoreFree tooling for signing and verifying software artifacts — a practical way to implement the SLSA provenance goal above.
49in-totoVerifies each step of a supply chain happened as expected — complementary to Sigstore's artifact-signing focus.
50CycloneDXA Software Bill of Materials (SBOM) format — a machine-readable list of every component in your app, for transparency and vulnerability tracking.
51SPDXAnother widely-used SBOM/license-declaration standard, often required by enterprise customers doing vendor security review.
52The Update FrameworkProtects software auto-update systems specifically against repository or signing-key compromise.
53CISA SBOMUS government guidance on why and how to produce a Software Bill of Materials — the policy layer behind the CycloneDX/SPDX formats above.

5 · Secure coding, assurance & testing programs

Organizational maturity models and professional testing standards — most relevant once you're past a solo project and building a team's security practice.

#ResourceWhat it's for
54CERT Secure Coding StandardsLanguage-specific secure-coding rules, especially strong for C/C++ and Java — useful if your stack includes either.
55BSIMMBenchmarks how mature a software-security initiative is by comparing it against real observed industry practices.
56SAFECodeIndustry-authored guidance on secure development practices and software assurance, similar spirit to SSDF.
57PTESA standard methodology for structuring an authorized penetration test end-to-end, from scoping to reporting.
58OSSTMMA rigorous methodology for testing operational security across systems, networks, and communication channels.
59CRESTAn accreditation body for professional penetration-testing, red-team, and incident-response providers — useful when hiring an external security firm.
60SANS InstituteWidely respected training and certification in penetration testing, SOC operations, and digital forensics.
61ISO/IEC 27001An internationally recognized information-security management system standard — referenced in Stage 14 as a common enterprise-customer requirement.
62BSI IT-GrundschutzA detailed German methodology covering organizational, system, and infrastructure security together.
63SOC 2 / AICPAAn audit standard covering security, availability, and confidentiality controls — the most common ask from US B2B SaaS customers, per Stage 14.
64PCI Security Standards CouncilOwns PCI DSS, the mandatory standard the moment you handle card data directly — Stage 14 recommends avoiding this scope via a processor like Razorpay/Stripe instead.
65Common CriteriaA formal, government-recognized security evaluation and certification scheme for IT products — heavier-weight than most startups need.
66OpenCREA tool for mapping the same requirement across multiple overlapping standards (e.g., "where does this ASVS item show up in ISO 27001?").

6 · Identity & protocol standards

The specifications underneath Stage 05's auth design — what a JWT, OAuth flow, or passkey actually is, defined by the bodies that standardized them.

#ResourceWhat it's for
67IETFPublishes the RFCs behind TLS, HTTP, and most internet/security protocols — the ultimate source of truth when a library's docs are ambiguous.
68OpenID FoundationStandardizes OpenID Connect, the identity-federation layer built on top of OAuth ("Sign in with Google" etc.).
69OpenID FAPIA stricter API security profile for high-value use cases like open banking — relevant if PixelGram-style app ever handles financial data.
70FIDO AllianceStandardizes passkeys/FIDO2 — phishing-resistant authentication, referenced in Stage 05 as the long-term upgrade path beyond SMS OTP.
71W3C WebAuthnThe actual browser API that implements FIDO2 passkeys on the web.
72OAuth Security Best Current PracticePractical security guidance for implementing OAuth correctly — most real-world OAuth vulnerabilities come from ignoring this document.
73SPIFFEA standard for giving services/workloads (not just humans) a verifiable identity — relevant in larger, zero-trust microservice architectures.
74CSAFA machine-readable format for publishing security advisories — mostly relevant if you're the one disclosing a vulnerability, not consuming one.
75OASIS OpenThe standards body behind CSAF and STIX/TAXII below, among others.
76STIX / TAXIIA standard format and exchange protocol for sharing structured cyber-threat intelligence between organizations.

7 · National & regional cyber guidance

Government-issued guidance, useful for confirming a decision against a recognized authority rather than just a vendor's opinion.

#ResourceWhat it's for
77UK NCSCClear, practical guidance on secure development, cloud, authentication, and incident management — often more approachable than US equivalents.
78ANSSIFrance's national cybersecurity agency, with strong system and architecture hardening guidance.
79Australian Cyber Security CentrePublishes the well-regarded "Essential Eight" mitigation strategies alongside broader hardening and incident-response guidance.
80ENISAThe EU's cybersecurity agency — threat-landscape reports and guidance spanning cloud, supply chain, and incident response.
81CERT-EUThreat intelligence and incident-response guidance for EU institutions, often useful reading regardless of jurisdiction.
82CISA Cybersecurity Best PracticesThe US equivalent — broad defensive practices and infrastructure-protection guidance.
83Mozilla Web Security GuidelinesVery practical, copy-pasteable guidance on TLS config, security headers, and cookies — the direct source behind several Stage 08/09 recommendations.
