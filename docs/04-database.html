04

Database — structure, scaling, and how it actually gets hacked

Get the schema and indexes right early — migrating a live table under load is the least fun afternoon in software. This stage also covers what happens once one database isn't enough, and how a database is actually attacked, step by step.

Relational by default

PostgreSQL is the right default for almost everything with users, relationships, and money-adjacent data. Reach for a document store (MongoDB) or a cache (Redis) only when you have a specific access pattern that relational modeling actively fights — not because it feels more modern.

Structure: index what you query by, constrain what must be unique

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  username  String   @unique
  passwordHash String
  createdAt DateTime @default(now())
}

model Follow {
  followerId  String
  followingId String
  @@id([followerId, followingId])   // prevents duplicate follows at the DB level
  @@index([followingId])            // fast "who follows me" lookups
}

Three structural decisions matter more than the rest: normalize until duplication would cause a real bug (a user's email lives in one place, not copied onto every post), index every column you filter or sort by in a hot query path, and put a uniqueness constraint at the database level for anything that must never collide — application-code checks alone have a race-condition window between "check" and "insert" that a real database constraint closes for free.

Migrations discipline

Every schema change is a migration file, checked into git, applied the same way in every environment (local → staging → production). Never hand-edit a production schema through a GUI — you lose the ability to reproduce it.

Scaling: what happens after one database isn't enough

Alex Xu's System Design Interview lays out the standard progression clearly (Chapter 1, "Scale From Zero To Millions Of Users") — the order below is that same progression, applied to the database tier specifically:

  1. Vertical scaling first — a bigger machine (more CPU/RAM/IOPS). Simplest option, and the right first move; it has a ceiling and a single point of failure, but it buys time without adding architectural complexity.
  2. Read replicas — one writable "leader" database and one or more read-only "follower" copies kept in sync via replication. Reads (which usually outnumber writes many times over on a social app's feed) get spread across followers; only writes hit the leader. If the leader goes down, a follower can be promoted — this is also your first real answer to "what if the database dies."
  3. Caching in front of the database — a cache-aside pattern (check Redis/Memcached first, fall through to Postgres on a miss, populate the cache after) removes repeat reads from the database entirely. An expiration policy (TTL) plus an eviction policy (commonly LRU) keeps the cache bounded and roughly fresh.
  4. Sharding (horizontal scaling) — once a single leader can't hold or serve the whole dataset, split the table's rows across multiple independent databases ("shards") using a sharding key (e.g. userId) that decides which shard a given row lives on. Every shard has the same schema; the data on each is different.
What sharding costs you (also straight from Chapter 1)

Sharding is a real technique, not a free upgrade. It introduces three concrete problems: resharding — what happens when one shard outgrows its capacity or a bad sharding key produces an uneven split, forcing you to move data and update the routing logic while live; the hotspot/celebrity problem — a poorly chosen key can dump disproportionate traffic on one shard (one extremely active user's data hammering a single shard while others sit idle); and cross-shard queries — a join or aggregate that used to be one query across one table now has to fan out across many shards and merge results in application code.

Consistent hashing (Chapter 5 of the same book) is the standard fix for the resharding problem specifically: instead of a naive hash(key) % N (where adding or removing one server reshuffles almost everything), servers and keys are placed on a conceptual ring by hash value, and each key belongs to the next server clockwise from it. Adding or removing a server then only affects its immediate neighbors on the ring, not the entire dataset — this is the same idea behind how a load balancer distributes requests and how a distributed cache decides which node holds which key.

Beyond a single relational database: how distributed stores stay correct

If you ever move part of your data into a distributed key-value store (Cassandra, DynamoDB, Riak — the subject of Chapter 6, "Design A Key-Value Store"), a few ideas from that world explain why those systems behave the way they do, and they're worth recognizing even if Postgres is all you use:

CAP theorem

During a network partition, a distributed system must choose between Consistency (every read sees the latest write) and Availability (every request gets a response). You can't fully have both at once — most key-value stores are tuned toward one side (CP or AP) and let you tune per-operation.

Quorum consensus (N/W/R)

Data is replicated to N nodes; a write must be acknowledged by W of them, a read must query R of them. Picking W+R > N guarantees every read sees at least one copy of the latest write — a tunable knob between consistency and latency.

Vector clocks

When two replicas accept conflicting writes during a partition, a vector clock records which version "happened before" another, so the system (or the app) can detect and resolve the conflict instead of silently picking one write and losing the other.

Gossip protocol & hinted handoff

Nodes tell each other which peers are up or down (gossip), so the system detects a failed node without a central coordinator. If a node is briefly unreachable, another node temporarily holds ("hints at") the writes meant for it and hands them off once it's back — a temporary failure doesn't become a permanent data loss.

The hacking side: how a database actually gets breached

Every step below is a real, common attack chain — not a hypothetical. Each one maps to a specific, fixable cause.

1. SQL injection — the classic

// Vulnerable: user input concatenated straight into the query string
const query = `SELECT * FROM users WHERE username = '${input}'`;
// An attacker submits:  ' OR '1'='1
// The query the database actually runs becomes:
//   SELECT * FROM users WHERE username = '' OR '1'='1'
// which matches every row in the table, not zero.

This is how it proceeds in a real attack: the attacker first tries a single quote in every input field and watches for a database error in the response — that error is confirmation the input reaches a raw query. From there they use UNION SELECT to pull columns from other tables (like a password_hash column no one meant to expose), then time-based or boolean-based blind techniques to extract data one bit at a time even when the app shows no visible error. The fix is not "sanitize the input" — it's structural: parameterized queries / an ORM that always sends user input as data, never as part of the SQL text, so there is no string for an attacker to break out of. Stage 03 and the worked example in Stage 18 both build on this.

2. Excessive privilege turns a small bug into a total breach

The injection above is only catastrophic if the database user the app connects as can read every table, drop tables, or read other databases on the same instance. A properly scoped, least-privilege app user — one that can only touch the specific tables/rows the app needs, with no DROP/ALTER permission — turns "attacker can run arbitrary SQL" into "attacker can still only see what the app itself could see," which is a much smaller blast radius. This is the single highest-leverage database security control that isn't about the query at all.

3. A database left reachable from the public internet

If a database has a public IP and its firewall allows inbound connections from anywhere, an attacker doesn't need an application bug at all — they scan for open database ports directly, then try default/leaked credentials. This is exactly why Stage 08/09 puts the database in a private subnet reachable only from the app tier's security group: there is no network path to attack from the outside, regardless of how the credentials leak.

4. Leaked credentials or a leaked backup

A database password committed to git, pasted into a support ticket, or sitting in an unencrypted backup file in a misconfigured public storage bucket is functionally the same breach as a successful injection attack — the attacker just connects directly with valid credentials. This is why Stage 05/13 insists secrets live in a secrets manager (never in source), and why backups need both encryption at rest and access controls at least as strict as the live database's.

5. NoSQL injection (the document-database version)

// Vulnerable: a MongoDB query built from raw, unvalidated request input
db.users.find({ username: req.body.username, password: req.body.password });
// An attacker sends password as an OBJECT instead of a string:
//   { "username": "alice", "password": { "$ne": null } }
// "$ne": null means "not equal to null" — true for any real password hash,
// so the query matches alice's account with no valid password at all.

The fix is the same principle as SQL injection: never let raw client input become part of a query's structure. A strict schema validator (Stage 03's zod example) that rejects a password field unless it's a plain string closes this before it ever reaches the database driver.

6. Lateral movement from a compromised app server

If an attacker compromises the app server itself (through an unrelated bug — a vulnerable dependency, an exposed debug endpoint), the database credentials sitting in that process's environment variables let them pivot straight to the database with full app-level access. Least-privilege DB users, short-lived/rotated credentials, and network segmentation (Stage 08/09) all limit what that pivot actually gains an attacker, which is why no single control on this list is sufficient alone — they're layered on purpose.

Security checklist — database
  • Parameterized queries / an ORM everywhere — never build a query by concatenating a string CWE-89
  • Strict schema validation on any field that reaches a query, including document-store queries — rejects operator-injection payloads like $ne/$where CWE-943
  • Least-privilege DB user for the app — cannot drop tables or read other databases on the instance CIS Benchmarks
  • Encrypt at rest, including backups/snapshots (most managed databases do this by default — verify it's actually on) NIST Cryptographic Standards
  • Automated backups with a tested restore procedure — an untested backup is a hope, not a plan NIST CSF · Recover
  • Database in a private subnet with no public IP; only the app tier's security group can reach it CSA Cloud Controls Matrix
  • Credentials in a secrets manager, rotated on a schedule — never in source, never in an unencrypted backup CWE-798
