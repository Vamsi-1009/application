The shell and the filesystem tree#

A shell (usually bash) is a program that reads the commands you type and runs them. Everything on a Linux system — files, folders, even devices — lives under one single tree starting at / (the root). There's no "C: drive" concept; an external disk just gets mounted somewhere inside that same tree.

  • / — the root of everything
  • /home/yourname (shorthand: ~) — your personal files
  • /etc — system-wide configuration files (this is where Nginx's config lives in Stage 09)
  • /var/log — log files
  • /usr/bin, /bin — where installed programs actually live

A path starting with / is absolute (always means the same place, no matter where you are). A path that doesn't is relative (means different things depending on your current folder). . means "here," .. means "one folder up."

Moving around and looking at things#

pwd                 # print working directory - "where am I?"
ls                  # list files in the current folder
ls -la              # -l = long format (permissions, size, date), -a = show hidden files (dotfiles)
cd /var/log         # change directory (absolute path)
cd ..               # go up one level
cd ~                # go home
cat file.txt        # print a whole file to the screen
less file.txt       # page through a file (q to quit) - use this for anything long
head -20 file.txt   # first 20 lines
tail -20 file.txt   # last 20 lines
tail -f app.log     # follow a file live as new lines are appended - the single most-used
                    # command while debugging a running service (see Stage 13)
man ls              # the full manual page for a command
ls --help           # a shorter built-in summary

Creating, copying, moving, deleting#

mkdir myproject           # create a folder
mkdir -p a/b/c            # create nested folders in one shot
touch notes.txt           # create an empty file (or update its timestamp)
cp source.txt backup.txt  # copy a file
cp -r sourcedir/ backup/  # copy a folder recursively
mv old-name.txt new-name.txt   # rename (mv is also how you rename)
mv file.txt ~/Documents/       # move into another folder
rm file.txt               # delete a file - no trash bin, this is permanent
rm -r somedir/             # delete a folder and everything in it
The command every beginner needs to fear a little

rm -rf / (or any careless rm -rf with a wrong path) deletes everything, permanently, with no confirmation and no undo. Before running rm -r on anything, run ls on the exact same path first and actually read the output. Never run rm -rf copy-pasted from the internet without understanding every character of the path.

Permissions and ownership#

Every file has an owner, a group, and three permission sets (owner / group / everyone else), each with read (r), write (w), and execute (x). ls -la shows this as a 10-character string like -rwxr-xr--.

chmod 600 .env           # owner: read+write only, nobody else - correct for secrets (Stage 06)
chmod 755 deploy.sh        # owner: read+write+execute, everyone else: read+execute - typical for a script
chmod +x script.sh         # add execute permission, keep everything else as-is
chown vamsi:vamsi file.txt # change the owner (and group) of a file
sudo some-command          # run one command as root ("superuser do") - only when you actually need to
Security checklist — permissions
  • Never chmod 777 anything to "make an error go away" — that grants read/write/execute to literally everyone on the system CIS Benchmarks
  • Secrets/config files (like .env) should be 600 — only the owner can even read them CWE-732
  • Run your application as a dedicated non-root user, using sudo only for the specific commands that need it — this is exactly Stage 09's "never run the app as root" rule DISA STIGs

Pipes, redirection, and combining commands#

The real power of a shell is chaining small, single-purpose commands together. A pipe (|) sends one command's output straight into the next command's input.

cat access.log | grep ERROR | sort | uniq -c
# read the file, keep only lines containing "ERROR", sort them,
# then collapse duplicates and count how many times each appears

command > file.txt    # redirect output to a file, overwriting it
command >> file.txt   # redirect output to a file, appending
command 2> errors.txt # redirect only error output
command &> all.txt    # redirect both normal and error output
find . -name "*.log" | xargs rm   # feed a list of results as arguments to another command

Searching and processing text#

grep "ERROR" app.log        # print lines containing "ERROR"
grep -i "error" app.log      # case-insensitive
grep -r "TODO" src/          # search recursively through a folder
grep -c "ERROR" app.log      # just count matching lines
sed 's/foo/bar/' file.txt    # replace the first "foo" with "bar" on each line
awk '{print $1}' access.log  # print just the first whitespace-separated field of each line
cut -d',' -f2 data.csv       # print the 2nd column of a comma-separated file
sort names.txt | uniq        # sort, then remove duplicate lines
wc -l file.txt                # count lines in a file

Processes and services#

ps aux              # list every running process
ps aux | grep node   # find a specific process
top                  # live, updating view of CPU/memory usage (q to quit)
kill 1234            # ask process 1234 to stop (graceful)
kill -9 1234         # force-kill it (last resort - no cleanup happens)
command &            # run a command in the background
jobs                  # list background jobs in this shell session
nohup command &       # keep a command running even after you disconnect
systemctl status pixelgram-backend    # check a service (Stage 09)
systemctl restart pixelgram-backend   # restart it
systemctl enable pixelgram-backend    # make it start automatically on boot

Networking from the terminal#

curl https://api.example.com/health       # make an HTTP request, print the response
curl -I https://example.com                # headers only
curl -sI https://example.com | grep -i server   # combine with grep, as used throughout this manual
ping example.com                            # check basic connectivity
ss -tulpn                                   # list listening ports (modern replacement for netstat)
ssh vamsi@203.0.113.5 -p 22022               # connect to a remote server (Stage 09)
scp file.txt vamsi@203.0.113.5:/home/vamsi/  # copy a file to a remote server over SSH

Installing software#

On Ubuntu/Debian (what this manual's VM stages assume), packages are installed via apt:

sudo apt update              # refresh the list of available package versions
sudo apt upgrade              # install available updates
sudo apt install nginx git    # install specific packages
apt list --upgradable         # see what's outdated without installing anything
sudo apt autoremove            # clean up packages nothing depends on anymore

Disk space and archives#

df -h                        # disk space free/used per mounted filesystem, human-readable
du -sh some-folder/           # total size of one folder
tar -czvf backup.tar.gz dir/  # compress a folder into a single .tar.gz archive
tar -xzvf backup.tar.gz       # extract one
rsync -avz src/ user@host:/dest/   # efficiently sync files/folders, including over SSH

Environment variables and shell config#

echo $HOME                  # print one environment variable
env                          # list all of them
export API_KEY=abc123        # set one for this shell session only
# For anything that should persist, add an export line to ~/.bashrc
# (interactive shells) - never hardcode real secrets there; use the
# secrets-manager pattern from Stage 06 instead.
echo 'alias ll="ls -la"' >> ~/.bashrc   # a shortcut you'll actually use daily

Scheduling recurring tasks#

crontab -e     # edit your personal scheduled jobs
crontab -l     # list them

# minute hour day-of-month month day-of-week   command
0 3 * * *   /home/vamsi/scripts/backup.sh   # every day at 3:00 AM - see Stage 14's backup discipline

Reading logs#

journalctl -u pixelgram-backend      # logs for one systemd service
journalctl -u pixelgram-backend -f   # follow them live
journalctl --since "1 hour ago"      # narrow by time
tail -f /var/log/nginx/access.log    # Nginx logs directly, as used in Stage 13

Security checklist — the terminal itself
  • Never pipe a downloaded script straight into a shell (curl ... | bash) without reading it first — you're handing a stranger's code full account access OWASP ASVS
  • Prefer sudo <single-command> over opening a permanent root shell — smaller window, clearer audit trail CIS Benchmarks
  • Double-check the exact path before any destructive command (rm -r, chmod -R) — there's no undo CWE-732

This is deliberately a reference, not an exhaustive shell course — come back to it whenever a command in an earlier or later stage looks unfamiliar.
