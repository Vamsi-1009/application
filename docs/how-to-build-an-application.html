Field Manual · 2026 Edition

Building an application, ground up

Everything between "I have an idea" and "it's live, it scales, and it doesn't leak" — frontend, backend, database, auth, email/OTP, secrets, storage, CDN, VMs, load balancers, domains, monitoring, and the security practice threaded through all of it.

Scope: web application, mobile-ready API Audience: first build → senior/staff level Region notes: India-specific compliance & providers called out inline Companion: full 103-item security resource catalog
00

How to use this manual

This is one continuous build, told in the order you actually do it. Each stage ends with a security checklist (amber) tying that stage to a named standard, and — where relevant — an India note (teal) on providers, compliance, or market conventions. Skim top to bottom the first time; come back to individual stages as a reference once you're building for real.

A rule that saves every stage below

Security is not a stage you do after building — it's a property you decide on while building, at the same moment you decide how a feature works. Every checklist below sits inside the stage it belongs to, not bolted on at the end, because that's the only place it's cheap to apply.

01

Plan & threat-model first

Before a line of code: what are you building, who is it for, and what happens if it's attacked or it fails?

Define the thing in one paragraph

Write down: who the user is, the one core action they take (post a photo, book a slot, send money), and what "done" looks like for v1. Everything you add beyond this is scope creep until v1 ships and has real users.

Sketch the architecture before the code

Browser/App  →  CDN (static assets, images)
                 │
                 ▼
             Load Balancer  →  App servers (VMs, autoscaled)
                                  │
                    ┌─────────────┼─────────────┐
                    ▼             ▼             ▼
               Database      Object storage   Email/OTP
               (Postgres)     (S3-like)        service
                    │
               Secrets manager (the "password wallet"
               for the app itself, not for end users)

Threat-model in five questions

  • What data would hurt someone if it leaked? (passwords, phone numbers, payment info, private messages)
  • Who can call each endpoint, and what stops them calling it for someone else's data?
  • What happens if this service gets 100x the traffic in an hour?
  • What happens if a dependency (npm package, cloud provider, email vendor) is compromised?
  • Who gets paged when something breaks at 2 a.m., and how do they know?
Security checklist — planning
  • Do a lightweight STRIDE or attacker-goal pass on your architecture sketch before writing code MITRE ATT&CK
  • Decide your data classification (public / internal / sensitive / secret) up front — it drives every later encryption and access decision NIST CSF
  • Write your incident-response contact and escalation path before you need it, not during an incident FIRST
02

Frontend

The part users touch. Optimize for clarity of state (loading / error / empty / success) before you optimize for visual polish.

Pick a stack and stop debating it

For 2026, a defensible default: React + TypeScript + Vite for a single-page app, or Next.js if you want server rendering and API routes in one codebase. TypeScript is not optional at this point — it catches an entire category of bugs (wrong shapes crossing the frontend/backend boundary) before a user ever sees them.

The four states every screen needs

  • Loading — a skeleton or spinner, never a blank white screen.
  • Empty — what a brand-new user sees with zero data; this is most people's first impression.
  • Error — a message that says what happened and what to do next, not a raw stack trace.
  • Success — the actual content, plus confirmation for anything destructive (delete, unfollow, cancel).

Where tokens live (this decision matters more than it looks)

// Access token: kept ONLY in memory (a module-level variable or React context).
// Never in localStorage/sessionStorage — any XSS on the page can read those.
let accessToken: string | null = null;

// Refresh token: never touched by JS at all. The backend sets it as an
// httpOnly + Secure + SameSite=Strict cookie, so a script on the page
// cannot read or exfiltrate it even if XSS occurs.
async function apiFetch(path: string, opts: RequestInit = {}) {
  const res = await fetch(path, { ...opts, credentials: "include",
    headers: { ...opts.headers, Authorization: `Bearer ${accessToken}` } });
  if (res.status === 401) {
    const r = await fetch("/api/auth/refresh", { method: "POST", credentials: "include" });
    if (r.ok) { accessToken = (await r.json()).accessToken; return apiFetch(path, opts); }
    window.location.href = "/login";
  }
  return res;
}
Security checklist — frontend
  • Never use dangerouslySetInnerHTML (or raw innerHTML) on user-generated text — rely on the framework's default escaping OWASP · XSS
  • Set a Content-Security-Policy header restricting script sources; it's your backstop when an escaping mistake still happens Mozilla Web Security
  • Treat every client-side validation rule as UX only — the backend must re-validate everything, because the client is attacker-controlled OWASP ASVS
  • Don't put secrets (API keys, anything sensitive) in frontend code — anything shipped to the browser is public CWE-798
  • Pin dependency versions and run npm audit / Dependabot in CI — the frontend has the deepest dependency tree in most apps OpenSSF
03

Backend & API

The part that enforces the rules the frontend merely suggests. Assume every request is hostile and check anyway.

Pick a stack

A defensible 2026 default: Node.js + TypeScript + Express (or Fastify/NestJS), or Python + FastAPI if your team leans Python. Either is fine — what matters far more than the framework is the discipline below.

Validate at the boundary, authorize on every route

// 1. Validate shape and reject unknown fields (zod, pydantic, whatever your
//    language offers) — never trust a request body's shape.
const CreatePost = z.object({ caption: z.string().max(2200), imageKey: z.string() }).strict();

// 2. Authorize: can THIS user do THIS action on THIS resource?
//    This is the single most commonly missed check in real breaches —
//    OWASP calls it Broken Object Level Authorization (BOLA/IDOR).
async function deletePost(req, res) {
  const post = await db.post.findUnique({ where: { id: req.params.id } });
  if (!post) return res.status(404).end();
  if (post.authorId !== req.user.id) return res.status(403).end(); // <- the check
  await db.post.delete({ where: { id: post.id } });
  res.status(204).end();
}

Rate-limit anything an attacker can script

Login, signup, OTP send/verify, and password reset are the endpoints attackers automate against. Rate-limit per account and per IP, and lock out or slow down after repeated failures rather than returning an unlimited number of guesses.

Good error handling

Log the real error internally with full detail; return a generic message externally ("Something went wrong") in production. Stack traces are a gift to an attacker mapping your internals.

Pagination, always

Any list endpoint (feed, comments, search) needs cursor- or offset-based pagination from day one — an unbounded query is a denial-of-service waiting for enough rows.

Security checklist — backend/API
  • Authorization check on every route that reads or writes a specific resource, not just authentication OWASP API Security · BOLA
  • Parameterized queries / ORM only — never string-concatenate SQL CWE-89 · SQL Injection
  • Rate-limit auth, OTP, and password-reset endpoints per account and per IP CWE-307
  • Explicit allow-list of JWT signing algorithms — never accept "none" or let the client choose the algorithm OWASP · Broken Authentication
  • Global request body size limits and per-route timeouts to blunt resource-exhaustion attacks OWASP API Security · Unrestricted Resource Consumption
  • Centralized error handler that never returns stack traces or internal identifiers in production responses OWASP ASVS
  • Server-Side Request Forgery guardrails on any endpoint that fetches a URL supplied by the client OWASP · SSRF

Database

Get the schema and indexes right early — migrating a live table under load is the least fun afternoon in software.

Relational by default

PostgreSQL is the right default for almost everything with users, relationships, and money-adjacent data. Reach for a document store (MongoDB) or a cache (Redis) only when you have a specific access pattern that relational modeling actively fights — not because it feels more modern.

Index what you query by, constrain what must be unique

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  username  String   @unique
  passwordHash String
  createdAt DateTime @default(now())
}

model Follow {
  followerId  String
  followingId String
  @@id([followerId, followingId])   // prevents duplicate follows at the DB level
  @@index([followingId])            // fast "who follows me" lookups
}
Migrations discipline

Every schema change is a migration file, checked into git, applied the same way in every environment (local → staging → production). Never hand-edit a production schema through a GUI — you lose the ability to reproduce it.

Security checklist — database
  • Encrypt at rest (most managed databases do this by default — verify it's actually on) NIST Cryptographic Standards
  • Least-privilege DB user for the app — it should not have permission to drop tables or read other databases on the instance CIS Benchmarks
  • Automated backups with a tested restore procedure — an untested backup is a hope, not a plan NIST CSF · Recover
  • Put the database in a private subnet with no public IP; only the app tier's security group can reach it CSA Cloud Controls Matrix

Auth, OTP & the password wallet

Two different "password wallets" get confused constantly — get the distinction straight before building either.

Two meanings of "password wallet" — pick the right one

1. End-user password management — your app hashing and verifying each user's own password, optionally with a personal vault feature (like a browser password manager) if your product actually needs one.

2. Application secrets management — your infrastructure's own credentials (DB password, JWT signing key, third-party API keys) held in a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager) instead of a .env file committed to git. Almost everyone building "an app with a password wallet" actually means this second one — most apps don't need a personal-vault feature at all.

Hashing a user's password correctly

import argon2 from "argon2";
// Argon2id: memory-hard, resistant to GPU-cracking — OWASP's current
// recommendation over bcrypt/scrypt for new systems.
const hash = await argon2.hash(plainPassword, { type: argon2.argon2id });
const valid = await argon2.verify(hash, attempt);

OTP (one-time password) for email/phone verification

import crypto from "crypto";
const code = crypto.randomInt(100000, 999999).toString(); // not Math.random()
const codeHash = await argon2.hash(code);   // store the hash, never the plaintext
// expire in 10 minutes, cap verify attempts at 5, cap resends at 3/hour/account
await db.otpCode.create({ data: { userId, codeHash, expiresAt: in10Min() } });

Sessions: short-lived access token + rotating refresh token

  • Access token (JWT): 15-minute expiry, used on every API call, never stored anywhere persistent.
  • Refresh token: long-lived, opaque random value, stored hashed in the DB, delivered via httpOnly cookie, rotated on every use — reusing an old refresh token should revoke the whole session chain (a strong signal of theft).

The application secrets manager, done right

// Boot-time: fail fast if a required secret is missing — never silently
// fall back to a default or an empty string in production.
const Secrets = z.object({
  DATABASE_URL: z.string().url(),
  JWT_SIGNING_KEY: z.string().min(32),
  SMTP_PASSWORD: z.string().min(1),
}).parse(process.env);
// In production these env vars are injected by the platform from
// AWS Secrets Manager / Vault at deploy time — never committed to git,
// never logged, rotated on a schedule.
Security checklist — auth & secrets
  • Argon2id (or bcrypt/scrypt as a fallback) for password hashing — never MD5/SHA-1/plain SHA-256 NIST Cryptographic Standards
  • OTP codes: cryptographically random, hashed at rest, short expiry, capped attempts and resends CWE-330 · CWE-307
  • Multi-factor / passkey support where the product warrants it — phishing-resistant beats SMS OTP long-term FIDO Alliance
  • No secret ever committed to git — use a secrets manager and a .gitignore'd .env locally with a `.env.example` for onboarding CWE-798
  • Least-privilege IAM: the app's runtime role can read only the specific secrets it needs, nothing account-wide CSA Cloud Controls Matrix
  • Rotate signing keys and long-lived credentials on a schedule, and immediately on suspected compromise NIST SSDF

Email / OTP delivery

Generating the OTP is the easy 10%. Getting the email to actually land in an inbox instead of spam is the other 90%.

Pick a transactional email provider

AWS SES, SendGrid, Postmark, or Resend are all reasonable — SES is the cheapest at scale and pairs naturally if the rest of your infra is on AWS. Whichever you pick, you're sending through their SMTP/API, not rolling your own mail server.

Deliverability is a DNS problem, not a code problem

  • SPF — declares which servers are allowed to send mail for your domain.
  • DKIM — cryptographically signs outgoing mail so receivers can verify it wasn't tampered with.
  • DMARC — tells receiving mail servers what to do with mail that fails SPF/DKIM, and gives you reports on abuse of your domain.

Skip any of the three and a meaningful fraction of your OTP emails land in spam — which, from a user's point of view, is indistinguishable from your OTP feature being broken.

Security checklist — email/OTP delivery
  • SPF + DKIM + DMARC configured on the sending domain before launch, not after users complain IETF RFCs
  • Never put the OTP code or a password-reset token verbatim in a URL query string that could end up in logs or a browser's history OWASP ASVS
  • Rate-limit OTP send and verify separately — sending is a cost/abuse vector, verifying is a brute-force vector CWE-307
  • Never reveal in an error message whether an email address exists in your system ("no account found" vs. generic "if this email exists…") OWASP · User Enumeration

File storage & the CDN

User-uploaded images/videos and your static frontend assets don't belong on the same server that runs your API.

Object storage + presigned uploads

Use S3 (or an S3-compatible store). The API never touches the image bytes — it hands the client a short-lived, scoped "presigned URL," the client uploads directly to storage, and the API just records the resulting object key.

// Backend: issue a scoped, time-limited upload URL — the object key is
// namespaced under the requesting user's own prefix so they can't
// overwrite or reference someone else's objects.
const key = `posts/${userId}/${crypto.randomUUID()}.jpg`;
const uploadUrl = await s3.getSignedUrl("putObject", { Bucket, Key: key, Expires: 300 });
res.json({ uploadUrl, objectKey: key });

// Client: PUT the file straight to storage — the API server never
// sees or holds the binary, which is both faster and removes an
// entire class of upload-based attacks from the API surface.
await fetch(uploadUrl, { method: "PUT", body: file });

Front the bucket with a CDN

CloudFront (AWS), Cloudflare, or Fastly sit between users and your storage bucket: they cache images at edge locations close to the user (faster loads), absorb traffic spikes without hitting your origin, and are the only thing with public read access — the bucket itself stays private.

Security checklist — storage & CDN
  • Bucket is private by default; only the CDN (via Origin Access Control) can read from it CIS Benchmarks
  • Validate file type and size server-side before issuing the presigned URL, and re-check content-type on the object once uploaded — never trust a client-declared MIME type alone OWASP · Unrestricted File Upload
  • Object keys scoped to the uploading user's own namespace, checked server-side, to prevent one user overwriting or referencing another's files (an IDOR variant) OWASP API Security · BOLA
  • HTTPS-only at the CDN edge; redirect any HTTP request Mozilla Web Security

Servers & the VM

No containers here by design — plain virtual machines, hardened directly, so the operating-system layer isn't a black box.

Provision, then harden, before you deploy anything

# Key-only SSH, no passwords, no root login
sudo sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo sed -i 's/^#\?PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config
sudo systemctl restart sshd

# Firewall: only what the app actually needs
sudo ufw default deny incoming
sudo ufw allow 22/tcp   # SSH (ideally restricted to a bastion/VPN IP range)
sudo ufw allow 443/tcp  # HTTPS via the load balancer / reverse proxy
sudo ufw enable

# Automatic security patches
sudo apt install unattended-upgrades -y

Run the app as a service, not a terminal you forgot open

# /etc/systemd/system/app-backend.service
[Unit]
Description=App backend
After=network.target

[Service]
User=appuser              # never root
ExecStart=/usr/bin/node /opt/app/dist/server.js
Restart=on-failure
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true

[Install]
WantedBy=multi-user.target

Put Nginx in front as a reverse proxy

Nginx terminates TLS, adds security headers, rate-limits at the edge, and forwards clean requests to your Node/Python process on localhost — your app process itself never listens directly on the public internet.

Security checklist — VM/server
  • SSH key-only, root login disabled, a firewall that denies by default CIS Benchmarks
  • App runs as an unprivileged, dedicated OS user, never root DISA STIGs
  • Automatic OS security patching enabled CIS Controls
  • TLS 1.2+ only, modern cipher suites, HSTS header from the reverse proxy Mozilla Web Security
  • Server identification headers (like Server: nginx/1.x) stripped or minimized so a scanner can't fingerprint your exact version CIS Benchmarks

Load balancers & autoscaling

The mechanism that turns "one VM that falls over on a busy Tuesday" into "traffic that just gets absorbed."

The load balancer

An Application Load Balancer (AWS ALB, or Nginx/HAProxy if self-managed) sits in front of multiple app VMs, terminates HTTPS, health-checks each instance, and routes traffic only to healthy ones — this is also how you deploy with zero downtime (roll instances one at a time, the LB stops sending traffic to whichever is being replaced).

Autoscaling — the actual "volume increaser"

An Auto Scaling Group watches a metric (CPU, request count, queue depth) and adds VMs when it's under load, removes them when it's quiet. This is the mechanism people mean when they say a system needs to "handle more volume" — it's not a single button, it's a policy: minimum instances, maximum instances, and the trigger metric.

# conceptually, an autoscaling policy:
min_instances: 2
max_instances: 6
scale_out_when: avg_cpu > 65% for 3 minutes
scale_in_when:  avg_cpu < 30% for 10 minutes
health_check: GET /health every 15s, 2 consecutive failures = unhealthy
Security checklist — load balancing/scaling
  • TLS terminates at the load balancer with a managed certificate that auto-renews — never a hand-rolled cert you'll forget to renew IETF RFCs
  • App VMs sit in private subnets, reachable only from the load balancer's security group — no direct public IP on app instances CSA Cloud Controls Matrix
  • Health checks on a dedicated /health path that doesn't leak internal state, so scaling decisions can't be spoofed by an attacker OWASP ASVS

Domain names & DNS

The address people type, and the records that tell the internet where to send them.

Buying and pointing a domain

  1. Register a domain through a registrar — GoDaddy India, BigRock, Namecheap, or Hostinger are all common choices for developers in India; a .in domain works fine for an India-first product, a generic .com/.dev for a global one.
  2. Delegate the domain's nameservers to your DNS provider (Route53, Cloudflare DNS) — this is the step that lets Terraform/your infra actually manage records.
  3. Point an A/ALIAS record at your load balancer for the API, and a separate record at your CDN distribution for the frontend/static assets.
  4. Issue an HTTPS certificate (AWS ACM, or Let's Encrypt via Certbot if self-managing Nginx) for every hostname you serve.
Security checklist — domain/DNS
  • Enable registrar-level domain lock/2FA — a stolen domain login is a full takeover of your product CIS Controls
  • DNSSEC where your registrar/DNS provider supports it, to prevent DNS spoofing IETF RFCs
  • Certificates on an auto-renewal path (ACM or Certbot cron) — an expired cert is a self-inflicted outage Mozilla Web Security

CI/CD pipelines

The automated path from "I pushed a commit" to "it's safely running in production."

  • CI (continuous integration): on every push — install deps, typecheck, lint, run the test suite, run a dependency-vulnerability scan.
  • CD (continuous deployment): on merge to main — build, run database migrations, deploy to a staging environment, run a smoke test, then roll out to production instances one at a time behind the load balancer.
# conceptual pipeline (GitHub Actions / GitLab CI, either works)
on: push
jobs:
  test:    npm ci && npm run typecheck && npm test
  audit:   npm audit --audit-level=high
  deploy:  # only on main, only if test+audit passed
           npm run build
           npx prisma migrate deploy
           ssh app-server "sudo systemctl restart app-backend"
           curl --fail https://api.example.com/health
Security checklist — CI/CD
  • Dependency vulnerability scanning wired into CI, failing the build on high-severity findings OSV / GitHub Advisory Database
  • Deploy credentials scoped narrowly and stored as CI secrets, never in the repo CWE-798
  • Build provenance — know that what's deployed is exactly what was reviewed and tested, not a hand-edited artifact SLSA

Monitoring & incident response

You find out about most real problems from a dashboard or an alert, not a user's angry email — if you've set it up.

  • Metrics: CPU/memory/request-rate/error-rate per instance, and business metrics (signups, OTP failures, upload failures).
  • Structured logs: JSON logs with a request ID that ties a user's request across every service it touched — and redact secrets/tokens/passwords from every log line.
  • Alerts: error-rate spikes, elevated latency, failed health checks, unusual auth-failure rates (a brute-force signal).
  • A written incident runbook: who gets paged, what the first three diagnostic steps are, and where the postmortem gets written afterward.
Security checklist — monitoring/IR
  • Redact Authorization headers, passwords, and OTP codes from all logs before they're ever written OWASP ASVS · Logging
  • Alert on abnormal authentication-failure rates — it's usually the earliest signal of credential stuffing MITRE ATT&CK · Credential Access
  • Keep an incident-response plan and a named contact path before you need either NIST CSF · Respond/Recover

Security hardening checklist — the consolidated view

Every amber box above, gathered in one pass, organized by domain instead of by build stage — use this as your pre-launch pass.

Identity & access

Argon2id hashing · short JWTs + rotating refresh · rate-limited auth/OTP · MFA/passkeys where warranted · least-privilege IAM everywhere

Injection & input

Parameterized queries only · strict schema validation rejecting unknown fields · output escaping, no raw HTML injection

Access control

Ownership checks on every mutating route · object keys scoped per user · no security decision made client-side alone

Transport & storage

TLS 1.2+ everywhere · encryption at rest for DB/storage · private subnets for DB and app tier · public-read only through the CDN

Secrets

Nothing in git · a real secrets manager · least-privilege read access · scheduled rotation

Supply chain

Dependency scanning in CI · pinned versions · build provenance · minimal third-party JS on the frontend

Operations

Redacted structured logs · alerting on anomalies · tested backups · a written incident runbook

Email/OTP integrity

SPF/DKIM/DMARC on the sending domain · hashed, expiring, attempt-capped OTPs · no user-enumeration in error messages

For a much deeper pass — mobile, AI/LLM features, IoT, industrial, automotive, and formal compliance frameworks — work through the full 103-item resource catalog (also mirrored in the appendix below), filtering to what's actually relevant to your product.

Compliance & governance, including India

Standards you adopt voluntarily for maturity, and rules you must follow because you operate in a given market.

General frameworks worth knowing

  • ISO/IEC 27001 — an information security management system; common ask from enterprise customers doing vendor due diligence.
  • SOC 2 — common ask from US-based B2B SaaS customers.
  • PCI DSS — mandatory the moment you touch card data directly (most teams avoid this entirely by using a payment processor like Razorpay/Stripe that keeps card data off your servers).
India-specific rules that actually apply to most apps

Digital Personal Data Protection Act, 2023 (DPDP Act) — India's core personal-data law. Practically: get explicit, specific consent before collecting personal data, state your purpose, let users withdraw consent and request deletion, and report significant breaches. Design your signup/consent flow and your account-deletion flow with this in mind from day one — retrofitting consent UX later is painful.

CERT-In directions (2022) — require reporting certain categories of cyber incidents to CERT-In within 6 hours of noticing them, and mandate that ICT system logs be maintained (typically referenced as a 180-day window) within Indian jurisdiction. If your monitoring/logging stack (Stage 12) is an afterthought, this is the rule that makes it not optional.

RBI guidelines — apply specifically if you touch payments/lending (tokenization requirements for card data, data-localization rules for certain payment data). Not relevant to a plain social app, essential if you add wallets or checkout.

Practical provider notes — AWS (Mumbai/ap-south-1) and GCP (Delhi/asia-south1) both have India regions if data residency matters to your compliance posture; Razorpay/Cashfree are the common payment-gateway choices; most teams use an Indian registrar (BigRock, GoDaddy India) for the domain and a global DNS/CDN provider (Route53, Cloudflare) for everything downstream.

Skill roadmap: junior → staff

What "more senior" actually means in this stack, stage by stage.

LevelWhat you ownSignal you've reached it
JuniorA single feature, in one part of the stack, with guidanceYou can implement a well-specified endpoint or screen correctly and test it
MidA full feature across frontend + backend + DB, unsupervisedYou catch your own authorization/edge-case bugs before review, not after
SeniorA subsystem (auth, feed, uploads) end-to-end, including its failure modesYou design for the case where a dependency is slow/down/compromised, not just the happy path
Staff/LeadThe whole architecture: how services, data, and infra fit together and scaleYou can explain a tradeoff (consistency vs. availability, cost vs. latency) and defend the choice you made
Principal/ArchitectCross-team technical direction and risk — security, compliance, cost, and scaling posture as one pictureOther senior engineers come to you before a decision, not after it's already made

The fastest way through this ladder isn't more frameworks — it's building one real system through every stage in this manual, then doing it again with the mistakes from the first one fixed.

Glossary

TermPlain-language meaning
VM (Virtual Machine)A rented slice of a physical server that behaves like its own computer — where your backend process actually runs.
Load balancerSits in front of several VMs and spreads incoming traffic across the healthy ones.
AutoscalingA policy that adds or removes VMs automatically based on how much traffic/load there currently is.
CDN (Content Delivery Network)A network of servers, geographically close to users, caching your static files and images so they load fast anywhere.
DNSThe system translating a domain name (example.com) into the server address it should connect to.
OTP (One-Time Password)A short code, valid once and briefly, used to prove someone controls an email/phone.
Secrets manager / "password wallet" (infra sense)A locked box for your app's own credentials (DB password, API keys) so they never sit in source code.
JWT (JSON Web Token)A signed, compact token proving who a logged-in user is, checked on every API request.
Reverse proxyA server (like Nginx) that sits in front of your app, handling TLS/headers/rate-limiting before forwarding clean requests inward.
IDOR / BOLAA bug where an app checks that you're logged in, but forgets to check whether the specific record you're accessing is actually yours.
